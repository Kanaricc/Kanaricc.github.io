Rust and OS

发表于 2020-01-17 更新于 2020-01-18 分类于 闲扯

rust是一门很有意思的语言,它的内存模型和0代价抽象等都很吸引人.几天速成想折腾写点好玩的东西.

rust的0代价抽象有些过分.一堆高阶函数,await和async也能做到0代价,和写汇编性能差不了多少,是有些服气.

严格来说这篇是读书笔记.

以C对撞Rust

自己主要还是C系语言用户,所以实际学习时,总是会相比较着进行.

变量声明

首先,rust也使用分号.每个变量的声明开头使用let,并且将变量类型跟随名称后面.

随着时间发展,现代语言几乎都抛弃了变量类型在左侧的声明方式.这大概是一个实践上的经验.左侧的类型声明会在复杂状况下造成极大的理解障碍.比如

阅读此类东西,总会是让人头疼.你会发现,为了理解它在干嘛,这种声明方式使得阅读顺序必须是顺时针的螺旋式,显得很不自然和明确.一旦调整类型到右侧时,这个问题似乎就不存在了.

所以不但变量的声明类型位于名称左侧,包括函数参数类型 返回值也都是位于右侧.

另外.

Type Refer
i8,i32,i64,i128 short,int,long long
u8,u32,u64,u128 unsigned
f32,f64 float,double
bool bool
char char

当然,rust也缺不了元组.

另外值得注意的是rust的char与str是utf8的,而不是ascii.所以在遍历时也不能按照c中的方式处理.

rust中的变量默认为不可更改.必须添加mut关键字才作为一般意义上的变量考虑.

控制流

rust里的if一类,没有小括号.(是不是有些像Go)

循环也有些不一样

最主要的是,多出了一个loop死循环,for循环以迭代方式完成.不过,其实还有一个重要的信息,它们都是表达式,能够返回值.

得益于rust的零代价抽象,迭代的使用相比直接的for(int i=1;i<=9;i++)几乎没有区别.

最后一行不带引号,则作为返回值返回.这样用if是允许的.(所以我们也没有三元运算符可用了)

函数声明

rust内的所有关键字都省略到了极简,也算是一种风格.

包管理

rust发展太快了,各种文章各说各话.这里主要是一个crate(一个包)内的结构.

rust使用mod来明确的标注一个模块.关键字默认为私有,使用pub来标注一个对外公开的关键字.在2018标准,文件也会作为一个模块,文件名作为模块名,文件内关键字同样默认私有.使用use关键字来显式的引入某个模块.

对于第三方模块,2018不再需要显式标注引入crate.(但是仍然需要use)

一个独立的rust

进行系统编程不同于应用级编程.一般情况我们并不会在意编程中的语言特性和系统特性的明确界限,但是这里,必须将它们剥离开,去掉系统特性.毕竟我们要编写OS.

  • 取消标准库

整个程序的启动流程大概实际上:由系统装载C运行时启动,C运行时调起rust运行时.rust运行时执行完成有关栈溢出保护 异常处理机制等操作后将控制权交给fn main.所以一旦取消了标准库,我们就要自行解决这些问题.

当取消了标准库时,我们就要被迫处理以下的东西.

  • 自定异常处理函数
  • 取消异常退栈操作
  • 自定程序入口
  • 取消C运行时依赖

除了代码中多了许多一般接触不到的细节,在cargo的配置中也有一些东西.

代码将panic置为了abort主要是因为退栈的操作.在普通编程中,abort将会让程序直接崩溃退出,内存交由系统释放;panic则是由程序完成内存释放.退栈是一个系统相关的机制,无法在OS层上完成,此处使用不带退栈的abort.

编译指令

CPU异常

CPU异常很常见,比如

  • 除0错误."把0个饼干分给0个朋友,每个朋友有多少个饼干,这种问题很没有意思,既没有饼干也没有朋友." --某人工智障助手SDELETEi
  • 内存页错误.访问了非法内存.
  • 非法操作符.程序执行了一个cpu不认识的机器码.
  • 一般保护性错误.
  • 二次错误.指第一次错误的恢复过程中又发生的错误
  • 三次错误.指第二次错误的恢复过程中又发生的错误
  • 四次错误没有套娃了.三次错误将会直接引起CPU重启.

一个需要强调的点是这些错误类型都是事先约定的,当异常发生时,CPU保证会从内存的指定位置读取中断表,将与错误相关的信息压入栈内,将程序运行转到中断表的处理函数代码位置,从此处开始执行.在这个过程中涉及的约定有

  • 中断表格式
  • 中断处理函数格式
  • CPU调用该函数的方式

中断处理函数和一般函数不一样.

寄存器与栈

寄存器有由调用者保存和被调用者保存两种,作为函数参数传递数据的寄存器也有固定的使用(rdi,rsi,rdx,rcx...),返回值默认用rax保存等等.实际上这种行为是完全语言层面的,进一步的,CSAPP中介绍的规范是C语言采用的.更详细的信息需要参见System V ABI(ABI: Application Binary Interface).

不知道是不是我书看得太草,感觉书里没说?现在如果有一门语言不遵守这种规范,那么一旦两门语言要链接在一起相互调用,就会产生不可预测的结果.这也是在C++中使用extern "C"的原因,这个标识会告知编译器使用C的规范生成结果.不巧的是,CPU中断的处理函数不遵守C规范(.

顺便,Rust目前没有属于自己的规范.另外,其实C++也没有,但是其编译器的实现有.

栈结构也是这样.不同编译器可能会给出有些不一样的使用策略.

为了生成符合中断处理函数规范的结果,需要注明extern "x86-interrupt",它实现了这个规范,不过同时也掩盖了很多细节.需要强调,不通过它,仅靠徒手也是可以完成的.
